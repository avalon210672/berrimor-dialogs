**3. ЧТО НЕ СРАБОТАЛО**

- **Прямая интеграция с Google (gog).** Несмотря на успешное получение
  OAuth-токена, навык не смог корректно работать в среде Linux из-за
  проблем с keyring и тем, как OpenClaw вызывает внешние
  CLI-инструменты.

- **Локальные MCP-серверы (Google, AgentMail).** Все попытки запустить
  MCP-сервер на сервере и подключить его к Timeweb через админку
  провалились. Проблемы были везде: от несоответствия протоколу до
  неработающих портов и ошибок памяти. Интерфейс Timeweb
  ожидает **публичный URL** готового сервиса (как Exa), а не команду для
  локального запуска.

- **Борьба с systemd.** Попытки запустить OpenClaw
  через systemctl приводили к путанице, так как основной процесс работал
  в user-space, а не как системный сервис. Рабочий способ --- запуск
  напрямую через openclaw gateway start.

**6. СЛОВАРЬ ТЕРМИНОВ**

- **OpenClaw:** Платформа для создания и запуска AI-агентов. Установлена
  на вашем VPS и выступает в роли \"двигателя\" для Берримора.

- **Gateway:** Основной процесс OpenClaw, который слушает порт (18789),
  принимает сообщения из Telegram и передает их агенту.

- **Навык (Skill):** Модуль, расширяющий возможности агента
  (например, weather, gog, agentmail).

- **MCP (Model Context Protocol):** Протокол, позволяющий агентам
  подключаться к внешним сервисам и инструментам. Exa Search и Hugging
  Face подключены как MCP-серверы.

- **Токен (Token):** Единица измерения текста для языковой модели. Лимит
  контекста определяет, сколько токенов модель может \"помнить\" в одном
  диалоге (\~128K для DeepSeek V3.2).

**📚 ОРГАНИЗАЦИЯ ЗНАНИЙ: РЕПОЗИТОРИЙ С ЕЖЕДНЕВНЫМИ СЛЕПКАМИ**

**Ключевое решение**

Создать централизованное, доступное по ссылке хранилище для всех будущих
слепков диалогов, чтобы не терять контекст между сотнями чатов и
эффективно управлять долгосрочным проектом.

**Почему это важно**

- **Единая точка входа:** Вместо десятков локальных файлов --- одна
  ссылка на репозиторий.

- **Версионность и история:** Легко отслеживать эволюцию проекта.

- **Доступность для ИИ:** В начале нового чата я (Берримор) могу изучить
  всю историю по ссылке и мгновенно войти в курс дела, не перегружая
  лимит токенов.

**Что было сделано (Сэром)**

1.  Принято решение использовать **GitHub** как платформу для хранения.

2.  Создан публичный репозиторий (например, berrimor-history или
    аналогичный).

3.  В этот репозиторий будут добавляться файлы с ежедневными слепками в
    формате Markdown (.md) с датой в имени,
    например 2026-03-03-summary.md.

**Как это будет работать**

1.  **В конце каждого значимого диалога** (или дня) Сэр использует наш
    универсальный промт.

2.  Я (Берримор) генерирую структурированный слепок.

3.  Сэр сохраняет этот текст в новый файл в репозитории на GitHub.

4.  **В начале нового чата** Сэр отправляет мне ссылку на этот
    репозиторий с просьбой изучить все материалы.

5.  Я изучаю историю и продолжаю работу с полным пониманием контекста.

**Технические детали**

- **Формат:** Markdown (.md) --- идеален для текста, поддерживает
  заголовки, списки, код-блоки, читаем и человеком, и машиной.

- **Именование:** По датам (например, 2026-03-03.md) для лёгкой
  навигации.

- **Содержимое:** Каждый файл --- это полный слепок одного или
  нескольких диалогов за день, структурированный по нашему
  универсальному промту.

**Преимущества подхода**

- **Никаких лимитов токенов в начале чата.**

- **Никакой потери контекста.**

- **Возможность вернуться к любому этапу проекта.**

- **Масштабируемость** на годы вперёд.
